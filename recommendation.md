For production deployment, keep the current architecture split (crawler, index/storage, API/UI) but run it under a supervised process model with explicit resource controls. Use a process manager (systemd, supervisord, or container orchestration) to manage lifecycle, restarts, and health checks. Keep SQLite in WAL mode only for small to medium workloads on a single machine; if ingestion and query volume grow, migrate storage to PostgreSQL while preserving the same logical schema (`pages`, `word_index`, `visited`, `crawl_sessions`, `failed_urls`). Put strict operational limits on crawl rate, queue depth, per-domain concurrency, and timeout budgets to avoid overload or accidental abuse. Add structured logs and metrics (crawl throughput, error rates, queue saturation, search latency p95/p99) and wire alerts for sustained failures or index lag.

To support reliable “search while indexer is active” at larger scale, evolve toward near-real-time indexing with a write-ahead ingest pipeline. Keep crawler discovery and fetch as one stage, then push parsed documents into an internal durable queue (still single-machine if needed, e.g., local disk queue). A dedicated index writer can batch commits for better write efficiency while exposing a committed checkpoint/version number to search APIs. The API can return both results and the current index checkpoint so clients understand freshness. This preserves the current behavior but makes consistency explicit, improves recovery after interruption, and provides a clean migration path if multi-machine scaling is needed later.
